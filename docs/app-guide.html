applybot

A local-first job application assistant — scrape, manage, autofill, and track without leaving your machine.

FastAPI backend Next.js dashboard Electron desktop shell Chrome extension Local-first & private

What is it?

applybot turns job-posting URLs into a structured, reviewable pipeline. You paste a URL, the backend scrapes and normalizes the listing, you pick which of your three saved profiles should apply, and a companion browser extension autofills the live form — all while the dashboard tracks every session and event.

Everything runs on your own machine. No cloud account is required, no resume or personal data ever leaves localhost unless you push it yourself.

📋

Job ingestion

Paste one or many URLs. The API scrapes HTML, detects the ATS (Greenhouse, Lever, or unknown), and stores structured data.

👤

Profile slots

Three named slots (profile_1 – profile_3) hold your contact info, work history, education, and parsed resume.

📎

Resume parsing

Upload a PDF resume into any slot. The backend extracts structured fields automatically.

🧩

Extension autofill

The Chrome extension fetches the active profile payload and fills the live application form field-by-field.

📊

Session tracking

Every autofill attempt is logged as an ApplicationSession with timestamped events and a final result.

🖥️

Dashboard

A dark control-room UI shows job intake metrics, profile state, extension readiness, and full session history.

Architecture

Four loosely coupled layers communicate over HTTP:

  • FastAPI backend (apps/api/) — the single source of truth. Handles scraping, profile storage, session lifecycle, and the extension config endpoint. Uses SQLAlchemy + Alembic on Postgres.
  • Next.js dashboard (apps/web/) — Pages-Router UI that proxies /api/* to the local backend at port 8000. Provides the job console, profile editor, and session viewer.
  • Electron shell (apps/desktop/) — optional packaging layer that opens the dashboard in a desktop window. Business logic lives in the API, not Electron.
  • Chrome extension — browser-side worker. Reads GET /extension/config to discover which profile is active, creates sessions, fills forms, and posts events back.

Prerequisites

  • Docker (for Postgres)
  • Python 3.12+
  • Node 18+
  • Chrome (for the extension)

Quick start

1 — Start infrastructure

# From the repo root
docker compose up -d

2 — Run the API

cd apps/api
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
alembic upgrade head
uvicorn app.main:app --reload --port 8000

The API seeds the three profile slots automatically on first run. Verify with GET http://localhost:8000/health.

3 — Run the dashboard

cd apps/web
npm install
npm run dev

Open http://localhost:3000. The dashboard proxies /api/* to the local API.

4 — (Optional) Run the desktop shell

cd apps/desktop
npm install
npm run dev

5 — Load the extension

  1. Open chrome://extensions
  2. Enable Developer mode
  3. Click Load unpacked and select the extension source folder
  4. The extension will call GET /extension/config to bootstrap

Core workflow

  1. Set up a profile
    Go to /profiles in the dashboard. Pick one of the three slots, fill in your contact info, work history, and education — or upload a PDF resume and let the parser pre-fill the fields.
  2. Activate a profile slot
    Mark one slot as active. The extension always reads whichever slot is currently active via /extension/config.
  3. Ingest a job
    Paste the job posting URL into the dashboard console on /jobs. The backend fetches the page, normalises the data, and stores it.
  4. Open the job posting in Chrome
    Navigate to the actual application form. The extension detects the page and shows its action button.
  5. Trigger autofill
    Click the extension button. It creates an ApplicationSession on the backend, retrieves the profile payload, and fills each form field. Fill events are logged as they happen.
  6. Review and submit manually
    Inspect the pre-filled form, correct anything the parser missed, and click Submit yourself. The extension records the final result.
  7. Review in the dashboard
    Check /sessions for the full event log and outcome of each application attempt.

Managing profiles

There are exactly three profile slots: profile_1, profile_2, profile_3. These are intentional — one for each identity or resume variant you might want (e.g., engineering, product, research).

Each slot holds:

  • Display name and contact details (email, phone, location, LinkedIn, GitHub)
  • Work experience entries
  • Education entries
  • An uploaded PDF resume (stored locally, never committed)
  • Parsed resume fields that can pre-populate the profile
Privacy: All resume files and profile data live under profiles/ which is gitignored. Nothing under that directory is ever committed.

Uploading a resume

Use the Upload Resume action on the profile page (or POST /profiles/{slot}/resume). After uploading, trigger Parse Resume (POST /profiles/{slot}/parse-resume) to extract structured fields automatically. You can then review and save the parsed data.

Adding jobs

Paste one or more job posting URLs into the ingestion console on the /jobs dashboard page. The API:

  1. Fetches the raw HTML for each URL
  2. Detects whether the ATS is Greenhouse, Lever, or unknown
  3. Normalises the posting into a structured Job record (title, company, description, ATS type)
  4. Stores the record in Postgres

You can also sync markdown job sources via the /watcher endpoint for file-based workflows.

Application sessions

An ApplicationSession is created when the extension begins filling a form. The lifecycle is:

  1. Extension calls POST /application-sessions (links job + active profile slot)
  2. Extension calls GET /application-sessions/{id}/payload to get flattened fill data
  3. Extension calls POST /application-sessions/{id}/events for each field it fills
  4. Extension calls POST /application-sessions/{id}/result with the final outcome

The dashboard /sessions page shows all sessions, their statuses, and the complete event timeline for each one.

Browser extension

The Chrome extension is the browser-side autofill worker. It communicates exclusively with the local API — there is no remote server involved.

Key behaviours:

  • On install / popup open, fetches GET /extension/config to learn the active profile slot and dashboard URL.
  • Only fills forms when triggered manually — no background auto-submit.
  • Reports field-level events so you can see exactly what was filled and what was skipped.
  • Final submission is always performed by the user, not the extension.
Terms of service: Automated form filling may conflict with some job platform terms. Use responsibly and review each application before submitting.

Next steps

The current v1 is intentionally minimal. Below are the most natural directions to extend the project.

  • Improve resume parsing accuracy
    The current parser does basic PDF text extraction. Integrating an LLM or a dedicated parsing library (e.g., pyresparser, Affinda) would yield much higher-quality structured output for experience and education fields.
  • Expand ATS coverage
    Only Greenhouse and Lever are currently detected. Adding Workday, Taleo, iCIMS, and BambooHR scrapers would significantly increase the range of supported job postings.
  • Cover letter generation
    Add a POST /jobs/{id}/cover-letter route that takes the active profile and job description and generates a draft via an LLM. Store the draft alongside the job record.
  • Job matching / scoring
    Embed job descriptions and profile summaries and compute similarity scores so the dashboard can rank which jobs best fit each profile slot.
  • Markdown job source watcher
    The /watcher endpoint stub exists. Wiring it to a file-system watcher that automatically ingests URLs from a local markdown file would enable a Obsidian / plain-text workflow.
  • Extension field-mapping editor
    Let users correct which profile field maps to which form selector. Corrections could be persisted and re-used on the same ATS in future sessions, improving fill accuracy over time.
  • Session analytics dashboard
    Build a dedicated analytics view: fill rate by ATS, most-missed fields, time-to-complete, and submission success rate. This would identify patterns across many applications.
  • Optional cloud sync (later)
    For users who want their data accessible from multiple machines, an optional encrypted sync layer (e.g., self-hosted S3, Supabase with RLS) could be layered on top without changing the local-first core.

Items in green are low-effort quality improvements. Blue items add net-new capabilities. Orange items improve the feedback loop over time. Grey items are intentionally deferred.
